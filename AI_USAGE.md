# AI Usage Notes

## Tools Used

- **Claude Sonnet 4.6**
- **ChatGPT 5.3** 

---

## What I Used AI For

### Initial project scaffold
I asked Claude to generate the FastAPI project structure — `database.py`, `models.py`, base Pydantic schemas, and route skeletons. This covered the mechanical parts of setting up a typed Python project so I could focus on TDD and business logic.

### Docker multi-stage build
Claude produced a starting Dockerfile structure. I reviewed it and added the non-root user (`adduser --system`), the `HEALTHCHECK` instruction, and the named `/data` volume for SQLite persistence — all of which were missing from the initial output.

### GitHub Actions workflows
Claude produced working CI and Docker build workflows. I adjusted the `--cov-fail-under` threshold and added the `upload-artifact` step for coverage reports.

### Rate limiter skeleton
I asked Claude for an in-memory sliding-window rate limiter. The initial version had no `reset()` method, which caused cross-test pollution — all tests sharing the same middleware instance would bleed request counts into each other. I identified the problem from failing tests, added the `reset()` method, and wired it into a `conftest.py` `autouse` fixture.

## Where I Overrode or Corrected AI Output

### lifespan parameter missing
Claude generated the `lifespan` async context manager correctly but forgot to pass it to the `FastAPI()` constructor (`lifespan=lifespan`). The application started without errors but tables were never created, causing `no such table: urls` on every request. Tests passed because they create their own in-memory DB via fixtures and never go through the lifespan event. I identified the root cause by reading the startup logs, confirmed the missing parameter, and added it.

### Timestamp handling
Claude initially used `default=datetime.utcnow` in the SQLAlchemy model — deprecated in Python 3.12. I replaced this with `server_default=func.now()` and `onupdate=func.now()` so timestamps are managed at the database level, not in application code.

### Collision handling strategy
Claude's first version of `create_short_url` used a pre-check query (`db.query(...).first()`) before inserting. This is a TOCTOU race condition — two concurrent requests could both pass the check and then collide on insert. I rewrote this to use `db.flush()` inside a try/except `IntegrityError` block, relying on the DB-level unique constraint as the source of truth.

### Test isolation — rate limiter
Claude did not account for the rate limiter being a singleton middleware instance shared across all tests. I added a `_get_rate_limiter()` helper that walks the middleware stack to find the `RateLimitMiddleware` instance, and an `autouse` fixture that calls `limiter.reset()` before each test.

### Test isolation — database
Claude initially suggested module-scoped DB fixtures. I changed this to function-scoped with explicit transaction rollback, giving true per-test isolation with no state leaking between tests.

### Schema structure
Claude generated a single `schemas.py` file. I split this into `schemas/url.py`, `schemas/stats.py`, and `schemas/health.py` — cleaner separation by domain, easier to extend.

### URL validation
Claude used `HttpUrl` as the Pydantic field type, which caused serialisation issues in response models. I switched to `str` with an explicit `field_validator` checking the URL scheme prefix.

### Missing files
Claude's initial `present_files` call omitted `requirements.txt`, `requirements-dev.txt`, `pytest.ini`, and `.gitignore`. I identified the missing files, requested them, and verified the full structure with `find` before proceeding.

### PYTHONPATH
Claude did not include `pythonpath = .` in `pytest.ini`, causing `ModuleNotFoundError: No module named 'app'` on a clean run. I fixed this in `pytest.ini` so the README instructions work as written.

## Code and Tests Written Without AI

- `tests/conftest.py` — the fixture design (function scope, transaction rollback, rate limiter reset via middleware stack walk) was written manually.
- All `assert` statements in tests — AI generated test function stubs but every assertion was written by hand to verify actual intended behaviour.
- `static/index.html` — written by hand; AI was not used for the frontend JavaScript or CSS.
- `app/constants.py` — identified the need for centralised error strings and wrote this independently.

---

## General Approach

I used AI as a **speed multiplier on structure**, not as a substitute for engineering judgment. Every file generated by AI was reviewed line by line before being used. Where AI output was incorrect, incomplete, or made assumptions I disagreed with, I rewrote it. The test suite was the primary quality gate — a failing or suspicious test meant fixing the code, not the test.
